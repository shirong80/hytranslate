# 정확성 게이트 참조 가이드

릴리스 노트가 실제로 출시되지 않는 기능을 광고하면 그것은 **버그**다. 노트 작성 후 게시 전에 모든 항목을 아래 게이트로 통과시킨다. 이 게이트의 목적은 "노트에 적힌 모든 것이 태그될 커밋에서 사용자에게 실제로 동작함"을 보장하는 것이다.

---

## 1. 태그될 커밋 기준으로만 기술한다

작업 트리의 미커밋/WIP 코드, 다른 브랜치의 기능은 **제외**한다. 사용자는 태그된 커밋으로 빌드된 산출물을 받기 때문이다.

### 의심 기능 검증 절차

어떤 기능이 정말 릴리스에 포함되는지 의심되면, 태그될 커밋본과 작업 트리를 직접 비교한다.

```bash
# 태그본(또는 태그될 커밋)에 해당 심볼이 존재하는가?
git show <tag-commit>:<path> | rg -i "<feature-symbol>"

# 작업 트리에만 존재하는가? (커밋 안 됨)
rg -i "<feature-symbol>" <path>
```

- 태그본에 **있으면** → 노트에 기술 가능.
- 태그본에 **없고** 작업 트리에만 있으면 → **WIP이므로 노트에서 제외**. 다음 릴리스로 이월.

### 실전 예시 (v0.1.0)

```bash
$ git show v0.1.0~0:src-tauri/src/window.rs | rg apply_fullscreen_overlay
# (출력 없음 → 태그 커밋에 없음)

$ rg apply_fullscreen_overlay src-tauri/src/window.rs
42:fn apply_fullscreen_overlay(window: &Window) { ... }
# (작업 트리에만 존재)
```
→ "전체화면 Space 위 표시" 기능은 작업 트리에만 있으므로 v0.1.0 노트에서 **제외**했다.

> `<tag-commit>`은 아직 태그를 만들기 전이면 `HEAD`(태그될 커밋)로 검증한다. 이미 태그가 있으면 `v0.1.0` 또는 `v0.1.0~0`을 쓴다.

---

## 2. PRD 로드맵이 아니라 구현된 코드만 근거로 한다

PRD/기획 문서의 "할 예정" 항목은 근거가 아니다. 사용자가 **실제로 도달 가능한지**를 코드에서 확인한다:

- **명령 핸들러**: Tauri `#[tauri::command]`, IPC 핸들러, API 라우트 등에 실제로 등록되어 있는가?
- **store/상태**: 상태가 실제로 읽고 쓰이며 UI에 반영되는가?
- **UI**: 버튼/메뉴/단축키가 실제로 바인딩되어 사용자가 트리거할 수 있는가?

세 곳 중 어디서도 도달 경로를 찾을 수 없으면 그 기능은 "구현됨"이 아니다.

---

## 3. 과장 금지 — stub/TODO/플래그 off 제외

모든 불릿은 코드 근거가 있어야 한다. 다음은 노트에 넣지 않는다:

- `// TODO`, `unimplemented!()`, `throw new Error("not implemented")` 등 stub
- 주석 처리된 코드
- 기능 플래그가 off로 빌드되는 코드 (`if (FEATURE_X_ENABLED)`가 false)
- 테스트에만 존재하고 프로덕션 경로에 없는 코드

확신이 없으면 그 항목은 빼는 쪽이 안전하다. 빠진 기능은 다음 릴리스에서 광고할 수 있지만, 없는 기능을 광고하면 즉시 신뢰를 잃는다.

---

## 4. 개인정보·내부 경로 미노출

노트와 예시에 다음을 넣지 않는다:

- 실제 사용자 데이터(`source_text`, `translated_text`, 사용자 입력 샘플 등)
- 내부 절대 경로(`/Users/<name>/...`), 토큰, 키, 내부 호스트명

예시가 필요하면 가공된 더미 값을 사용한다.

---

## 5. 이중 언어 정합

`새로운 기능`(한국어)과 `Changelog`(영어)가 1:1로 대응하는지 확인한다:

- 항목 **수**가 같은가?
- 항목 **순서**가 같은가?
- 각 항목이 **같은 의미**를 담는가? (한쪽에만 있는 디테일 금지)

빠르게 점검하려면 양쪽 번호 목록의 길이를 세고, 같은 번호끼리 의미를 대조한다.

---

## 게이트 통과 판정

위 5개 항목을 모두 통과한 노트만 게시한다. 통과하지 못한 항목은:
1. 코드 근거를 찾아 보강하거나,
2. 노트에서 제거하고 제외 사유를 기록한다(출력의 "정확성 게이트 결과"에 명시).
