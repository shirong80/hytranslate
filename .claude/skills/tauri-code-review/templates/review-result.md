# 리뷰 결과 출력 템플릿

리뷰 결과를 작성할 때 아래 형식을 그대로 채운다. `{...}`는 플레이스홀더다.

---

```
📊 리뷰 요약

| 심각도 | 건수 |
|--------|------|
| 🔴 P0 치명적 | {N}건 |
| 🟠 P1 잠재적 | {N}건 |
| 🟡 P2 개선 | {N}건 |

🔐 보안 핵심 축 점검
- ① IPC 커맨드 입력 검증: {통과 / 위험 / 확인 필요} — {한 줄 근거}
- ② 권한·스코프 최소화: {통과 / 위험 / 확인 필요} — {한 줄 근거}

✅ 계획 대비 정합성: {충족 / 부분 충족 / 미충족} — {한 줄 요약}
```

---

```
🔴 P0: 치명적 이슈

[P0-1] {이슈 제목}
- 📁 파일: `{파일경로}`
- 📍 위치: `{라인 번호 / 커맨드명 / capability 키}`
- 🔍 현재 코드/설정:
  {문제가 되는 코드 또는 JSON 발췌}
- ❗ 문제: {구체적 설명 — 가능하면 "웹뷰의 악성 JS가 ~할 수 있다"는 공격 시나리오로}
- 💡 수정 제안:
  {권장 코드 또는 설정}
- 📖 근거: {왜 위험한지 기술적 설명}
```

P1, P2도 동일 형식으로 작성한다. P0·P1에는 반드시 `💡 수정 제안`을 포함한다. P2는 제안이 권고 수준이면 생략 가능.

이슈가 없는 심각도 구간은 "해당 없음"으로 표기한다.

---

```
🔎 확인 필요 사항

| # | 질문 | 관련 파일 | 이유 |
|---|------|-----------|------|
| 1 | {질문} | {파일:라인 또는 (미수신)} | {왜 확인이 필요한지} |
```
- diff/설정만으로 단정할 수 없는 항목, 또는 받지 못한 설정 파일(capabilities/tauri.conf 등)로 인해 판단을 보류한 영역을 여기에 기재한다.

---

```
🧭 총평

{전체 변경에 대한 2~3문장 종합 평가. 특히 보안 두 축(IPC 입력 검증, 권한·스코프)의 통과 여부를 명시한다.}

머지 가부: {승인 / 수정 후 재리뷰 / 수정 필수}
```
- P0가 1건이라도 있으면 "수정 필수"를 기본으로 한다.
- 보안 축 중 하나라도 "위험"이면 머지를 보류하는 방향으로 권고한다.
